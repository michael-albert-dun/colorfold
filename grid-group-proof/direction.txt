# Some symmetry

Imagine picking up the puzzle and giving it a quarter-turn clockwise before beginning and then applying the moves ADad -- in the sense of "upper left, lower right, upper right backwards, lower right backwards" but using "upper right" in the new position. You'll still cycle three elements but it will be the tree elements 4, 5 and 6 rather than 2, 5 and 8. If instead of rotating the puzzle you'd flipped it over (did I tell you it was two-sided?) it would still be the positions 2, 5 and 8 that you changed, but you'd actually cycle them in the reverse order.

# All the threes

An interesting thing about a three-cycle (x y z) is that if you apply it twice you get (x z y), and if you apply it three times of course you get back to ''nothing happens''. But what this means is that if we can work out somehow to produce a three-cycle that moves three specific positions, then we also have *every* three cycle that moves just those three positions (since x has to go to y or z, and that determines the third one).

But, so far we only have the three-cycles that move the central row or central column. Can we get more? Yes -- this is a standard trick. Imagine we do some moves, then we do the ADad sequence (cycling the middle row) and then we undo the moves we originally did (do them again in reverse order interchanging the Aa, Bb, Cc and Dd pairs). # Three's a crowd

Triples are nice, but it turns out that we can't achieve *all* permutations using just three-cycles (don't ask). But two-cycles (usually called *transpositions*), that just exchange two elements are a different story. Imagine we could use all the transpositions. That's a pretty easy puzzle to solve -- rearrange the grid below so that 1-9 are in the correct order.

{{Demo, clicking two tiles interchanges them - initial layout random, with a restart to a new random config}}

The reason this is easy is that there are any number of good strategies: switch 1 with whatever is in its place, then switch 2 with whatever is in its place, ...

So we'd like to move just two elements, but the basic operations we have (the corner rotations) move four, and we've concentrated on getting others that move three. It's time, I'm afraid for a bit of abstract thinking -- and first I want to think about the other direction: suppose that I knew how to move triples and pairs, how could I move four elements using just one pair? So, imagine we have four elements w, x, y, and z. Let's give ourselves the pair (y z), any triples we like, and try to get (w x y z). Can we find (some triple) (y z) to produce (w x y z)? Our triple should send w to x (which will be left alone by (y z). It should send x to z so that when we then apply (y z) it will go to y where we want it.  So let's look at (w x z) (y z). We've set it up so that this sends w to x and x to y. Where does it send y - the first part leaves it alone so it goes to z. And where does it send z - to w in the first part and then it's left alone, so we get exactly (w x y z).

{{I'd like a demo of this but I can't think of a good way to show it}}

So we've got

(w x z) (y z) = (w x y z)

If we undo the (w x z) on the left with (z x w) we get

(y z) = (z x w) (w x z) (y x) = (z x w) (w x y z)

and now we can plug in some numbers we know work (like (w x y z) = (1 2 5 4)) to produce (for now) the (5 4) transposition. Or, we could just search for three-cycles that combine with A to produce a transposition.

## We're done?

We haven't got a recipe, but we know that we can somehow exchange any pair of positions (and leave all the rest fixed) and that's that enough to achieve any rearrangement of the positions. That deserves to be called a theorem:

Theorem: Every rearrangement of the cells of a 3x3 grid can be achieved using rotations of its 2x2 subgrids.

But wasn't the problem supposed to be about 4x4 grids? Well yes, but remember the third secret (and think a little more).

Think about the 4x4 grid as four overlapping 3x3 grids. Surely we can use our power there to boost it up to the 4x4 case? In lots of cases we could just do it directly, but there are some irritating edge cases. If we want to move position 1 to the final row or final column, we can't just use the top left corner. But we can exchange 1 with any position that lies in all four grids and then we could exchange that with any other position. Finally, we could redo the original exchange and that gets the exchange of 1 with any other position. And we can do similar things for all the edge squares. So once again, we can do all exchanges and hence all permutations.

And why stop at 4? For 5x5 we have overlapping 4x4 grids and then the same trick. For 6x6, overlapping 5x5's etc.

Another approach would be to notice that being able to do adjacent exchanges is enough to do all exchanges and then that in any size bigger than 3x3 we can achieve each adjacent exchange by doing it within a 3x3 subgrid. But that's probably enough.
